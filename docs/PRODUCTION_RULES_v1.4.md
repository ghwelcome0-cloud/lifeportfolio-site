## 🟥 대원칙-A. 고유성은 종합하되, 표현은 직관적으로 딱 하나로 (2단 구조)

> **이것은 인생포트폴리오 전 영역 카피라이팅의 최상위 헌법이다. 모든 RULE-REPORT / RULE-PROGRAM / RULE-CAREER 규칙은 이 원칙을 위배할 수 없다.**

### A-1. 문제 정의 — 두 요구의 충돌
- 인생포트폴리오는 동시에 두 가지를 만족해야 한다:
  - **(가) 고유성**: 80억 인구 중 단 한 명(Only One). 56문항 응답을 빠짐없이 종합해야 한다.
  - **(나) 직관성**: Tesla·Nike·IKEA처럼 한눈에 꽂히는 글로벌 대기업급 압축 카피.
- 이 둘은 표면적으로 충돌한다. 고유성을 다 담으면 장황해지고(직관성 상실), 압축하면 고유성이 1/N로 붕괴한다.

### A-2. 해법 — 2단 구조 (Headline + Detail)
- **모든 핵심 카피는 2단으로 분리한다:**
  1. **1단 — 헤드라인(직관)**: 응답을 종합한 결과를 **단 하나의 직관적 이미지/문장**으로 압축. 큰 글씨. 쉼표 최소, 사족 금지. (브랜드 슬로건 수준)
  2. **2단 — 디테일/서브라인(고유성)**: 그 헤드라인을 **응답 기반으로 종합한 고유 근거**로 풀어 씀. 작은 글씨.
  3. (선택) **3단 — 푸터(근거)**: 어떤 응답(Q번호/키워드)에서 도출됐는지 표기.
- **핵심**: "종합은 디테일에서, 표현은 헤드라인에서". 고유성을 버리지 않고도 직관성을 얻는다.

### A-3. 적용 현황 (v1.4 기준)
| 영역 | 1단(헤드라인·직관) | 2단(디테일·고유성) | 구현 |
|---|---|---|---|
| 사명 | missionHeadline (압축 통찰) | missionSubline (응답 종합) | report-engine-v4.js, report.html |
| 비전 | visionHeadline | visionSubline | 동일 |
| 고유한 결 | toneLabel (응답기반 2300종) | toneTagline (응답기반 변주 풀) | program-engine.js, program-rules.json |
| 사명이 여는 길 | 직업명 3개 (CareerEngine) | 도메인 근거 | program-engine.js, career-engine.js |

### A-4. 신규 카피 제작 시 의무 체크
- [ ] 헤드라인이 **한 호흡**에 읽히는가? (쉼표 1개 이내 권장)
- [ ] 헤드라인이 응답을 **종합한 단일 이미지**인가? (나열·접속사 금지)
- [ ] 디테일이 **응답 기반 고유성**을 담는가? (톤 고정 문구 단독 금지)
- [ ] 같은 톤 사용자끼리 디테일이 **다르게** 나오는가? (fingerprint 변주 필수)

## 🟥 대원칙-B. 축적 원칙 — 개선은 누적, trade-off 금지

> **하나를 개선하면 하나를 잃는 것이 아니라, 계속 축적되어야 한다.**

### B-1. 정의
- 모든 개선은 **기존 품질을 보존(회귀 안전)하면서 새 품질을 더한다.**
- "A를 좋게 만들었더니 B가 나빠졌다"는 결과는 **규칙 위반**이다.

### B-2. 구현 강제 사항
1. **폴백 보존**: 새 변주 풀/엔진이 부재(구버전 캐시, 데이터 누락)하거나 fingerprint 미가용일 때 **반드시 기존 출력으로 폴백**한다. (예: `taglinePool` 없으면 기존 `tagline`, CareerEngine 미연결 시 기존 tonePack 직업)
2. **재현성 보존**: 같은 응답(fingerprint) → 항상 같은 결과. 개선이 재현성을 깨지 않는다.
3. **회귀 테스트**: 개선 후 기존 케이스(예: 김영식 모범본)가 동일 또는 향상 수준을 유지하는지 확인.
4. **버전 누적**: 규칙서는 이전 버전의 규칙을 삭제하지 않고 보강·승계한다. (v1.0→v1.2→v1.4)

### B-3. 위반 예시 (금지)
- ❌ tagline 변주를 넣으면서 기존 단일 tagline을 삭제 → 구버전 캐시 사용자 출력 깨짐
- ❌ CareerEngine 연결하면서 폴백 제거 → careerRules 로딩 실패 시 직업란 공백

**모범 출력 기준 (Reference Output)**:
- 자동 제작본: `docs/reference/REF_REPORT_KYS_2026-05-06.pdf`
- 자동 제작본: `docs/reference/REF_PROGRAM_KYS_2026-05-06.pdf`
- 본 두 PDF는 v1.2 규칙대로 합성된 결과의 **정식 모범 기준**이며, 향후 모든 출력은 이 결과물과 동일 수준의 정합성·고유성·자연성을 충족해야 한다.

### R6. 한국어 표현 자연성 (v1.2 신설, PR#66)
- **헤드라인 폴백 주어**: 도메인 정규화 미매칭 시 `'자기 자리에 있는 사람'` 사용 (이전 `'지금 살아가는 사람'`).
- **주격 조사 자동 보정**: 주어 끝 음절 받침 검사로 `이/가` 자동 선택. 받침 있음→`이`, 없음→`가`.
- **목적어·주어 중복 어법 차단**: `'사람을 ~하는 사람'` 패턴은 `'~으로 사람을 ~ 내는 사람'` 어순으로 자연화.
  - 예: `'사람을 원칙으로 지키는 사람'` → `'원칙으로 사람을 지켜 내는 사람'`
- **검증 항목**: grep으로 어색 패턴(`사람을 [원칙·마음·신념]으로 지키는 사람`, `지금 살아가는 사람이 사람을`) 잔류 0건.

### R7. 톤 SSOT (v1.2 신설, PR#65)
- 톤 결정의 단일 진실 소스(Single Source of Truth):
  ```
  resolveTone(scores, valueCats)  ← 가중치 합산 모델 (Q13 vc:+3, top1 axis:+2, top2 axis:+1)
    → report.toneKey                  [최상위 노출, 다운스트림 표준 입력]
    → report._v4Meta.toneResolution   [상세 근거 메타]
    → report.tone.key                 [본문 합성용 정정값]
  ```
- ProgramEngine.pickTone()은 0순위로 `report._v4Meta.toneResolution.toneKey`를 신뢰한다.
- v4 미가용 시에만 폴백 경로(report.tone → MV Compass → pragmatic 우세) 동작.

### R8. 사명/비전 응답 직합성 (v1.3 신설, RESPONSE-DIRECT)
- 사명/비전은 `synthMissionVisionFromResponses()` (report-engine-v4.js)가 고객 응답(Q13 가치·Q63 판단기준·Q75 분야·Q39/Q41 활동·Q73 성취)을 **100% 직접 합성**해 헤드라인 한 문장으로 만든다.
- **과거 7-slot / 3-Tier 템플릿 골격(headline/subline/diaryMission 3겹, "…자리에서, …나침반 삼아" 류)은 폐기**한다. 이 합성 함수가 구식 골격을 덮어쓴다(overwrite).
- 출력 필드:
  - `missionHeadline` / `visionHeadline` = 화면 노출 본문. 사명은 `…한다`, 비전은 `…된다/…살아간다`로 종결.
  - `missionSubline` / `visionSubline` / `footer` = 근거 안내문구("🔍 활동 응답…"). **데이터에만 보존, 화면 미노출**(R9 참조).
  - `diaryMission` / `diaryVision` = 빈 문자열(미사용).
- 메타: `report._v4Meta.missionVisionSource === 'response-direct'`로 합성 경로를 표시한다.
- **고유성**: 동일 입력 → 동일 결과(결정적). 80억 명이 진행해도 각자의 응답에서만 합성된다. 두 사람이 실제로 비슷한 답을 고르면 비슷한 결과가 나오는 것이 정상이다.

### R9. 중복 제거 — 화면 노출 최소화 (v1.3 신설, PR-중복제거)
- **근거 안내문구 미노출**: `🔍 활동 응답…` 류 subline/footer는 report.html에서 렌더하지 않는다(데이터로만 보존).

- 검증: 빌드 산출물 grep에서 `당신의 사명:` / `당신의 비전:` / `🔍 활동 응답`(화면 영역) 잔류 0건.

### R10. 자모·조사 가드 강화 (v1.3 신설)
- **자모 결합 금지**: 종결 처리 중 한글 음절과 낱자 자모(ㄴ 등)가 분리 결합되는 버그 차단. 예: `더하ㄴ다`(X) → `더한다`(O).

- 검증: 빌드 산출물에서 낱자 자모(ㄱ-ㅎ 단독) 인접 0건, `자유이 살아` 류 비문 0건.

- **다양성**: 개선 전 0.3% → 개선 후 **87.7%**(1500명). 화면 노출: `program.html` `.week-subline`.

### P7. 다음 단계 제안 = 1년 사이클 완주 이후 가이드 (v1.3 재정의, PR-#5)
- **재정의 배경**: 기존 nextSteps는 `1개월 후 / 3개월 후 / 1년 후` 시점에 본문(3주·3개월·1년) 일정을 그대로 재나열 → 본문과 중복이었다. 이를 폐기한다.
- **새 정의**: 프로그램(3주→3개월→1년)은 닫힌 일정표가 아니라, 한 사이클을 마치면 그 결과물이 다음 사이클의 출발점이 되는 **나선형 성장(spiral growth)** 구조다. nextSteps는 *1년 사이클을 완주한 이후*의 가이드 2문장만 제시한다.
  - `[{ when: "이 사이클을 마치면", task: <공통 메시지> }, { when: "다음 사이클은 이렇게", task: <고유 합성> }]`
  - 둘째 문장은 `primaryDomain`(Q75) × `userWeakGrain`(약축결) × `visionHeadline`로 응답마다 고유 합성한다.
- **영어 모드**: 엔진이 `opts.lang === "en"`일 때 when/task를 처음부터 영어로 직접 생성(`"When this cycle ends"` / `"For the next cycle"`). program.html의 `whenLabel` i18n에도 신규 라벨 매핑을 등록.
- **구현 위치**: program-engine.js `§6 다음 단계 제안` 블록. 구식 `L3_NEXTSTEPS_KO/EN`은 dead code로 보존(롤백 대비).
- 검증: 6 응답자 산출물에서 nextSteps 2개, 둘째 task의 도메인·약축결이 서로 다름(고유성). `1개월 후`/`3개월 후` 시점 라벨 0건.
- **[v1.4 보강, 대원칙-A]**: ① 첫 항목("이 사이클을 마치면")이 완전 고정이던 것을 `userTopStrength × primaryDomain × visionHeadline` 종합 변주 풀(6종, `variantIdx(0x7F11)`)로 전환. ② 둘째 항목 도입 문구도 변주 풀(4종, `variantIdx(0x7F23)`)로 표현 다양화(의미 보존). `_fixJosaPairs` 조사 보정(따옴표 뒤 `'통찰력'으로` 정확). 응답 부재 시 기존 고정 문장 폴백(**대원칙-B**). 다양성 3.5% → **37.6%**(약축 4종 구조 한계 내 최대화).

### P8. 고유한 결(typeLine) 2단 구조 — 응답기반 변주 (v1.4 신설, 대원칙-A 적용)
- **구조**: `{name}님의 고유한 결 — {toneLabel} · {toneTagline}` (program-engine.js `cover.typeLine`)
  - **1단 toneLabel(직관·응답기반)**: `report.tone.label`을 SSOT로 계승(리포트가 확정한 약 2300종 라벨). 이미 응답기반.
  - **2단 toneTagline(고유성·응답기반 변주)**: `tones[tone].taglinePool`(의미 보존 변주 8개) 중 `variantIdx(0x7A6C)`로 결정 선택.
- **문제(개선 전)**: toneTagline이 `tones[tone].tagline` 단일 고정값 → 같은 톤 사용자끼리 100% 동일(고유성 훼손). 사용자가 "새로고침/재생성해도 같은 결과"로 체감한 부분.
- **해법(v1.4)**:
  - `data/program-rules.json`: 5개 톤 각각 `taglinePool`/`taglinePool_en`(8개씩, **의미 방향 보존** — 톤의 정체성을 유지하되 표현만 다양화) 추가.
  - `program-engine.js`: `taglinePool`이 있으면 fingerprint로 1개 선택, 없으면 기존 `tagline` 폴백(**대원칙-B 축적 보장**).
- **검증**: 같은 톤(visionary_creator) 다른 응답 4건 → toneTagline 다양화 확인. 같은 fingerprint 2회 → 동일(재현성). fingerprint 없음(구버전 캐시) → 폴백 정상.
- **주의**: 기존 프로그램은 RTDB 캐시 → **재생성 시 적용**된다. (신규 생성은 program-loading.html에서 즉시 적용)

### P9. 응답기반 직업 매칭 연결 — CareerEngine (v1.4 신설, 대원칙-A/B 적용)
- **배경(ROOT CAUSE)**: `newPaths`(사명이 여는 길)·`fitJob`(직무 적합성)·`expansion`(직업 확장성)이 `tonePack`의 **톤 고정값**에서 나왔다. CareerEngine은 존재했으나 program-engine에 **연결되지 않았음**.
- **해법**: program-engine.js `build()`에서 CareerEngine를 호출(`ce` 블록)하여 응답기반 직업을 산출하고, newPaths/fitJob/expansion에 주입.
  - factory가 `root`를 받아 `root.CareerEngine` 참조(브라우저/Node 양쪽 지원).
  - `program-loading.html`(신규 생성)·`program.html`(재생성) 모두 `career-rules.json`·`mapping.json`을 로드해 `build({ careerRules, mapping })`로 전달.
- **효과(검증)**: 동일인의 응답에 따라 **운동선수→종목코치 / 배우→다큐디렉터 / 사업가→VC / 연구자 / 간호사 / 입법보좌관** 등 서로 다른 직업이 매칭됨. ("어떤 선수가 응답에 따라 배우·사업가도 될 수 있게" 요구 충족)
- **축적(대원칙-B)**: `careerRules`/`mapping`/CareerEngine 부재 시 기존 `tonePack`의 newPaths/fitJob/expansion으로 폴백(비파괴).
- **josa 가드**: 직업명 끝 음절 받침 검사(`_hangulJong`)로 `~로/~으로` 자동 선택. (예: "B-Corp 창업가로", "교사·강사로")

## RULE-CAREER v1.4 — 진로·교육 매핑 규칙

### C1. 13대 영역 5종 분리 (subType)
모든 영역에 동일한 5종 분류를 적용한다.

| subType | 정의 | Q3 강점 매핑 후보 |
|---|---|---|
| `practitioner` | 현장 실무·전문가 | 손기술, 실행력, 책임감, 헌신 |
| `researcher` | 연구·이론·분석 | 분석력, 통찰력, 학구열, 정직 |
| `business` | 사업·경영·창업 | 사업감각, 실행력, 도전, 리더십 |
| `media` | 콘텐츠·저널·평론 | 표현력, 창의력, 공감, 스토리텔링 |
| `policy` | 정책·행정·공공 | 분석력, 정의, 책임, 공정 |

### C2. 13대 영역 풀 (data/career-rules.json)
- 영역: 정치, 경제, 사회, 문화, 교육, 의료, 복지, 환경, 예술, 미디어, 스포츠, 법률, 종교 (13개).
- 각 영역 × 5 subType = 65개 진로 풀.
- 각 subType당 careers 4개 + education 3개.

### C3. 3축 결합 매트릭스
```
careers[0]   = primaryDomain × subType (Q3 강점 → 5종 중 1개 선택)
careers[1]   = primaryDomain × secondaryDomain (영역 융합, 의무)
careers[2]   = primaryDomain × Q41 열정 (열정 결합)
education[0] = subType별 전용 교육 풀 (단기, ≤3개월)
education[1] = primaryDomain × Q41 결합 교육 (중기, 6~12개월)
education[2] = secondaryDomain 보강 교육 (장기, ≥1년)
```

### C4. 강점 → subType 결정 알고리즘 (v1.1 + Q1 가중치 1.5)
```javascript
function pickSubType(q1Job, q3Strengths, q13Values, q41Topic) {
  var score = { practitioner:0, researcher:0, business:0, media:0, policy:0 };
  // Q1 직무 부분 매칭 (가중치 1.5)
  Object.keys(Q1_JOB_HINT).forEach(function(kw){
    if (q1Job && q1Job.indexOf(kw) !== -1) {
      Q1_JOB_HINT[kw].forEach(function(t){ score[t] += 1.5; });
    }
  });
  q3Strengths.forEach(function(s){
    SUBTYPE_KEYWORDS[s] && SUBTYPE_KEYWORDS[s].forEach(function(t){ score[t] += 1; });
  });
  q13Values.forEach(function(v){
    SUBTYPE_VALUES[v] && SUBTYPE_VALUES[v].forEach(function(t){ score[t] += 0.5; });
  });
  if (TOPIC_SUBTYPE_HINT[q41Topic]) {
    TOPIC_SUBTYPE_HINT[q41Topic].forEach(function(t){ score[t] += 1; });
  }
  return Object.keys(score).reduce(function(a,b){ return score[a]>=score[b]?a:b; });
}
```

### M0-1. 두 가지 수동 작업 방법 (v1.3 신설)

운영자는 회원 1인의 출력물을 만들 때 다음 **두 가지 방법 중 하나**를 선택한다. 두 방법 모두 동일한 v1.2 규칙·130점 루브릭을 적용하며, 산출물 품질은 동등해야 한다.

#### 방법 ① — AI 일괄 매칭 방식 (권장 · 빠름)
**개념**: 응답 데이터(스프레드시트 1행) + 제작 규칙서 v1.2 + 김영식 모범 PDF 2종을 **AI에게 한 번에 전달**하여, 사이트 자동 합성과 동일 수준의 결과를 받아 PDF로 출력·이메일 전송.

**절차 (3단계)**

```
[1단계] 입력 묶음 준비 (5분)
  └─ A. 응답 데이터: responses 탭에서 회원 1행 추출 (80개 컬럼 그대로)
        - CSV 또는 표 형태로 복사
        - Q1·Q3·Q13·Q41·Q47·Q49·Q63·Q73·Q75·Q79·Q80 핵심 슬롯 표시
  └─ B. 규칙서: docs/PRODUCTION_RULES_v1.2.md 전체
  └─ C. 모범 PDF 2종: docs/reference/REF_REPORT_KYS_2026-05-06.pdf,
                      docs/reference/REF_PROGRAM_KYS_2026-05-06.pdf
  └─ D. 매핑표: 본 문서 §C13 (13×5 매핑표) + §부록 A (영역별 상세 풀)

[2단계] AI에게 일괄 전달 (10분)
  └─ 프롬프트 골격 (그대로 사용):
     ─────────────────────────────────────────────────────
     첨부한 ① 회원 응답 데이터, ② 제작 규칙서 v1.2,
     ③ 김영식 모범 PDF 2종을 기준으로,

     본문을 모범 PDF와 동일한 구조·톤·고유성·자연성으로 합성하라.

     출력은 다음 12+6 슬롯의 정형 데이터로 반환할 것:
     [REPORT] 1.표지 2.사명 3.비전 4.강점3 5.성장3 6.진로3
              7.교육3 8.compass 9.primaryDomain 10.secondaryDomain
              11.subType 12.toneKey
     [PROGRAM] 1.분기테마 2.3주루틴 3.3개월 4.1년 5.부스터 6.성공기준·리스크

     R6(한국어 자연성)·R7(톤 SSOT)·C10(융합형)·C11(교육 차등)
     P4(미치환 토큰 0)·P6(톤 정합) 모두 충족하라.
     ─────────────────────────────────────────────────────

[3단계] 검증·발행 (10분)
  └─ M3 운영자 체크리스트 7항목 확인 (R1·R6·R7·P4·C10·C11·M1)
  └─ 130점 루브릭 자가 채점 → 120점 이상이면 PDF 변환·발송
  └─ 파일명: 최신_인생포트폴리오_[성명]_[YYYY-MM-DD].pdf
            최신_[성명]님_맞춤_실행_프로그램_[YYYY_MM-DD].pdf
  └─ 회원 이메일로 두 PDF 첨부 발송
  └─ RTDB 마킹: programs/{uid}/{sid}.manualProgramStatus = "manual_ai"
```

**장점**: 빠름 (총 25분), 사이트 자동 합성과 동일 톤·구조, AI가 매핑·자연어 보정 자동 수행.
**주의**: AI 출력은 반드시 운영자가 §M3 체크리스트로 검토 후 발행. 자동 합성과 100% 동일하지 않을 수 있으므로 모범 PDF와 좌우 비교 필수.

#### 방법 ② — 단계별 수기 매핑 방식 (정밀 · 컨트롤)
**개념**: 운영자가 직접 Word·PowerPoint 템플릿에 자리표시자를 채우고 PDF로 변환. 회원이 카운슬링·코칭을 신청했거나 사명·비전 라인을 자기 언어로 다듬어 달라고 요청한 경우, 또는 응답 일부 결손·특수 직군에 권장.

**절차 (5단계)** — §M1 표준 절차와 동일하게 진행 (아래 §M1 참조)
1. 응답 확인 (responses 탭 1행 검색·복사)
2. Word 매핑 (`template_report.docx` 자리표시자 기입, R1~R7 체크)
3. PowerPoint 작성 (`template_program.pptx` 3주/3개월/1년, P1~P6 체크)
4. PDF 변환 (파일명 규칙 준수)
5. 이메일 전달 (RTDB `manualProgramStatus = "manual"` 마킹)

**장점**: 운영자 자유 편집 5개 슬롯(M2)에 회원 맥락 정밀 반영 가능, 특수 직군·결손 응답에 강함.
**주의**: 시간 소요 (총 1~2시간), 자연어 보정·고유성 보장은 운영자 직접 책임.
