## 0.2 인바디(InBody) 비유 — 제품 구조를 이해하는 가장 쉬운 그림

```
   인바디 검사지            →   인바디 기반 맞춤 운동프로그램
   ───────────                  ──────────────────────
   인생포트폴리오 리포트     →   맞춤 실행프로그램
   (지금 나를 진단)              (그래서 무엇을 할지)
```

- **리포트 = 검사지:** 응답 데이터로 "지금 당신은 이런 사람"을 정밀하게 보여 준다.
- **프로그램 = 운동프로그램:** 그 진단을 근거로 "그래서 3주/3개월/1년 동안 이렇게 하라"를 준다.
- 둘은 **같은 톤(SSOT)** 으로 묶여야 하며, 프로그램은 반드시 리포트를 근거로 한다.

## 0.3 자산화의 큰 구조 (★ 사업의 전체 지도)

> 이것이 인생포트폴리오가 향하는 **최종 그림**이다. 리포트·프로그램은 그 첫 두 단계일 뿐이다.

```
[1] 리포트 + 맞춤 실행프로그램         ← 발견 + (살아낼) 설계
        │   "지금 나"를 진단하고, 무엇을 할지 받는다
        ▼
[2] 코칭 + 다이어리                    ← 살아냄 (실행의 지속)
        │   고객이 실제로 실행을 이어 간다. 코칭이 곁을 지키고,
        │   다이어리가 살아낸 흔적을 기록한다.
        ▼
[3] 방향별 커뮤니티                    ← 남김 + 연결성
        │   각 고객의 '방향'에 맞는 커뮤니티가 형성된다.
        │   같은 결을 가진 사람들이 서로의 자산을 연결한다.
        ▼
[4] 인생 자산화                        ← 가치 ↑ · 연결성 ↑
        한 사람의 인생이 자산이 되고,
        자산이 되면 가치가 높아지고 다른 자산과의 연결성도 높아진다.
```

**단계별 설명:**

| 단계 | 무엇을 | 성경적 대응 | 제품/기능 |
|------|--------|-------------|-----------|
| 1. 진단·설계 | 리포트 + 프로그램 | 발견 | 본 규칙서 제4·5부 |
| 2. 실행 지속 | 코칭 + 다이어리 | 살아냄 | (확장 로드맵) |
| 3. 연결 | 방향별 커뮤니티 | 남김 | (확장 로드맵) |
| 4. 자산화 | 가치·연결성 상승 | 인생 자산화 | 전체의 도달점 |

> **빌더가 기억할 것:** 리포트/프로그램을 만들 때도 항상 이 큰 구조를 의식한다. 리포트의 "다음 단계", 프로그램의 "축적 사이클(§6)"은 모두 [2]→[3]→[4]로 이어지는 **나선형 성장**의 입구다. 결과물은 닫힌 문서가 아니라 다음 사이클의 출발점이다.

# 제1부 · 미션과 철학 — 왜 만드는가

## 1.1 한 문장 미션

> **인생포트폴리오는 한 사람의 인생을 자산화한다.**
> 자산이 되면 그 사람의 가치가 높아지고, 다른 자산과의 연결성도 높아진다.

## 1.2 발견 → 살아냄 → 남김 → 인생 자산화

```
 발견          →   살아냄           →   남김            →   인생 자산화
 진단 응답         실행 프로그램         열매 / 결과물        가치 · 연결성 ↑
```

- **누구나 따라 할 수 있다. 그러나 진짜로 따라 하면 하나님 말씀대로 살고 열매를 맺는다.**
- **80억 인구 — 최소 국내 5,000만 명 — 각자의 고유성(Only One)을 보장하는 그 도달점까지 품질을 높인다.**

## 1.3 의미경제(Meaning Economy) 관점

기술(AI)이 효율을 맡는 시대에, 인간의 차별성은 **의미를 창조하는 능력**에서 나온다. 인생포트폴리오는 평범한 개인이 자기 인생의 의미를 **데이터로 발견하고, 실행으로 살아내고, 결과물로 남겨** 자산 생산자가 되도록 돕는다.

## 1.4 성경적 설계 기반

| 구절 | 의미 | 제품 반영 |
|------|------|-----------|
| 마태복음 24:45 | "때를 따라 양식을 나눠 줄 충성되고 지혜로운 종" | 각 사람에게 **맞춤(때를 따라)** 양식 — 고유성 |
| 마태복음 28:18–20 | "제자로 삼아 … 가르쳐 지키게 하라" | 발견에 그치지 않고 **살아내게(지키게)** 한다 — 프로그램·코칭 |

> **흔들릴 때 돌아올 곳:** 어떤 기능을 더할지 말지 판단이 안 설 때 → "이것이 한 사람의 고유성을 더 살리고, 자산화에 더 가까이 가게 하는가?"를 물어라. 답이 '아니오'면 하지 않는다.

# 제2부 · 두 대원칙 + 9가지 핵심 원칙 (헌법)

## 2.1 대원칙-A 「고유성 종합 × 직관적 단일표현 (2단 구조)」 — 최우선

> **"고유성은 종합하되, 표현은 직관적으로 딱 하나로."**
> **"종합은 디테일에서, 표현은 헤드라인에서."**

### 문제 (개선 전)
표현이 골격(고정 템플릿)에 갇혀 있었다. 같은 톤·분야 사용자끼리 **단일 고정 문구가 100% 동일** → 80억의 고유성이 표현 단계에서 사라짐. "새로고침해도 같은 결과"로 체감.

### 해법 (2단/3단 구조)
| 단 | 역할 | 성격 |
|----|------|------|
| **1단 헤드라인** | 직관 압축 (브랜드 슬로건 수준) | 누가 봐도 즉시 이해 |
| **2단 디테일/서브라인** | 응답 종합 고유성 | 같은 톤이라도 응답마다 다르게 |
| **3단 푸터** | 근거 Q번호 (선택) | 신뢰의 근거 |

### 2단 구조 적용 영역 (현황)
| 영역 | 1단 헤드라인(직관) | 2단 디테일(응답 종합 고유성) |
|------|-------------------|------------------------------|
| 사명/비전 | missionHeadline · visionHeadline | 응답 직합성 (Q13·Q63·Q75·Q73) |
| 고유한 결 | toneLabel (약 2,300종) | toneTagline 변주 8종 (variantIdx 0x7A6C) |
| 사명이 여는 길 | CareerEngine newPaths | 응답 기반 직업 매칭 |
| 분기 테마 | Quarter Heading (톤×Compass) | subline 8종 (variantIdx 0x5131) |
| 3주 루틴 | 주차 제목 | 주차 의미별 subline 6종×3 (0x6201+i) |
| 다음 단계 | "이 사이클을 마치면" | 변주 6종+도입 4종 (0x7F11/0x7F23) |

> **모든 2단은 fingerprint(=hash(응답)) 기반 결정 선택** → 같은 응답=같은 결과(재현성), 응답 부재 시 기존 출력 폴백(대원칙-B).

## 2.2 대원칙-B 「축적 원칙」 — Trade-off 금지

> **"개선은 누적한다. 어제의 사용자가 본 결과는 오늘도 유효해야 하고, 오늘의 개선은 내일 더 정교해진다."**

| 항목 | 내용 |
|------|------|
| **B-1 폴백 보존** | 풀 부재 / fingerprint 없음 → 기존 출력 그대로. 새 기능이 옛 기능을 깨지 않는다. |
| **B-2 재현성** | 같은 응답 = 같은 결과. fingerprint 결정 선택. 같은 fingerprint 2회 → 동일 산출(회귀 테스트 의무). |
| **B-3 버전 누적** | 옛 규칙 삭제 금지. v1.0→v1.2→v1.4→v2.0. 구식 로직은 dead code로 보존(롤백 대비). |
| **검증 게이트(4중)** | ① 폴백 ② 재현성 ③ 회귀 테스트 ④ 버전 누적 — 모두 통과해야 머지. |

## 2.3 9가지 핵심 원칙

| # | 원칙 | 한 줄 정의 |
|---|------|-----------|
| 01 | **응답 데이터 100% 우선** | 56문항 응답에서 직접 합성 |
| 02 | **고유성·개별성 최적화** | 1/N → 80억 인구 확장 보장 |
| 03 | **토큰 치환 완결성** | 미치환 `{{…}}` 0개 |
| 04 | **단일 진실 소스(SSOT)** | 톤·진로 결정 로직 일원화 (`resolveTone()`) |
| 05 | **한국어 표현 자연성** | 어색 결합 0건 · 조사 자동보정 |
| 06 | **응답 직합성** | 고정 골격 없이 응답→문장 합성 |
| 07 | **평이체 (P31)** | 명사형 정지어 → 동사형 진행어 |
| 08 | **2단 구조 (대원칙-A)** | 고유성 종합 × 직관 단일표현 (최우선) |
| 09 | **축적 원칙 (대원칙-B)** | 개선은 누적 · trade-off 금지 |

# 제3부 · 데이터 입력 규격 (56문항)

## 3.1 원재료 — 혼합형 검사지 v2

리포트·프로그램의 모든 문장은 **고객의 56문항 응답**에서 합성된다. (혼합형 V2 인생포트폴리오 검사, 총 11개 영역)

- 입력 형식: 검사 응답 결과 (스프레드시트 XLSX 또는 서비스 내부 응답 JSON).
- **원칙 01:** 응답 데이터 100% 우선. 톤 어휘와 충돌 시 **응답이 우선**한다.

## 3.2 1차 소스 문항 (가장 중요 — 직합성 핵심)

| Q번호 | 역할 |
|-------|------|
| **Q1** | 직무 가중치 (subType 결정 시 ×1.5) |
| **Q3** | 강점 |
| **Q13** | 가치 |
| **Q41** | 열정 (careers[2] 결합) |
| **Q63** | 사명/비전 합성 소스 |
| **Q73** | 사명/비전 합성 소스 |
| **Q75** | 1순위 관심 분야(primaryDomain) — 진로 매핑 핵심 |

> 사명/비전은 Q13·Q63·Q75·Q73을 종합해 **7-slot 템플릿 골격 없이 직접 합성**한다(R8).

## 3.3 13대 영역 (진로 매핑 축)

```
정치 · 경제 · 사회 · 문화 · 교육 · 의료 · 복지 · 환경 · 예술 · 미디어 · 스포츠 · 법률 · 종교
```

각 영역 × 5종 subType = **65 진로 풀** (제6부 참조).

# 제4부 · 리포트 제작 규칙 (10단 구조)

> **리포트 = 인바디 검사지.** "지금 당신은 이런 사람"을 응답 데이터로 정밀하게 보여 준다.

## 4.1 10단 고정 구조 (📘 ~ 🧪) — 순서·아이콘 불변

| # | 섹션 | 아이콘 | 핵심 내용 |

# 제6부 · 진로/교육 매핑 엔진 (CareerEngine)

## 6.1 핵심 사상 — 한 사람을 여러 직업으로

> **같은 '스포츠' 응답이라도 → 운동선수 / 데이터 분석가 / 마케팅 전략가 / 캐스터 / 체육 행정가.**
> 응답의 미세한 차이가 다른 직업을 만든다. 이것이 고유성의 진로 버전.

## 6.2 13대 영역 × 5종 subType = 65 진로 풀

| subType | 의미 |
|---------|------|
| practitioner | 현장 실무 |
| researcher | 연구·분석 |
| business | 사업·경영 |
| media | 콘텐츠·평론 |
| policy | 정책·공공 |

→ 13영역 × 5 = **65 진로 풀**.

## 6.3 3축 결합 산출 규칙

```
careers[0] = 영역 × subType            (주 진로)
careers[1] = 영역 융합 (의무, C10)      (단일 도메인 연속 차단)
careers[2] = Q41 열정 결합              (열정 반영)
```

## 6.4 subType 결정 공식

```
subType = argmax( Q1 직무 가중치 ×1.5 + Q3 강점 + Q13 가치 + Q41 열정 )
```

## 6.5 CareerEngine 연결 (P9)

- `build()`에서 CareerEngine 호출 → Q1·Q3·Q13·Q41·Q75 응답으로 직업 산출 → newPaths·fitJob·expansion에 주입.
- **축적(B):** CareerEngine 부재 시 tonePack 폴백(비파괴).
- **융합 의무(C10):** careers[1]은 반드시 영역 융합형. 단일 도메인 연속 차단.

# 제7부 · 합성 엔진 구현 디테일

> **엔진은 규칙 기반 합성(NOT LLM).** 같은 응답엔 항상 같은 답 — 신뢰 가능한 진단. 이것이 9,900원 진단의 품질 보증 근거다.

## 7.1 fingerprint — 고유성의 씨앗

```
fingerprint = hash(응답 전체)
```
- 같은 응답 → 같은 fingerprint → 같은 결과(재현성, B-2).
- 응답이 없으면 fingerprint 없음 → 변주 idx 0 반환(회귀 안전, B-1).

## 7.2 variantIdx(salt) — 결정적 변주 선택

> 같은 톤·분야라도 응답마다 다른 문장을 고르되, 같은 응답엔 항상 같은 문장.

```
variantIdx(salt):
   x = fingerprint XOR (salt × 0x9E3779B1)   // 황금비 상수로 bit-mixing
   x = xorshift(x)                            // 비트 섞기
   return x % pool.length                     // 풀에서 결정적 선택
   (fingerprint 없음 → 0 반환)
```

**salt 배정표 (영역별 고유 salt — 충돌 방지):**
| salt | 영역 |
|------|------|
| 0x7A6C | 고유한 결 toneTagline |
| 0x5131 | 분기 테마 subline |
| 0x6201+i | 3주 루틴 주차별 subline |
| 0x7F11 | 다음 단계 첫 항목 |
| 0x7F23 | 다음 단계 둘째 항목 도입 |

## 7.3 _fixJosaPairs — 조사 자동 보정

> 한국어 자연성(원칙 05). 받침 유무에 따라 조사를 자동 선택.

대상 조사쌍: **을/를 · 이/가 · 와/과 · 은/는 · 으로/로**

```
_jongBefore(text, pos):
   따옴표·괄호를 건너뛰고 직전 '한글 음절'을 찾아 받침 유무 판정
```
- 예) `'통찰력'으로` ✓ / `'아이디어'로` ✓
- 자모 가드(R10): `더하ㄴ다 → 더한다`, `창조와 자유이 → 자유가`.

## 7.4 tpl() — 토큰 치환

- compassKw·primaryDomain·userTopStrength 등은 **토큰**으로 보관 → 출력 직전 `tpl()`로 일괄 치환.
- **완결성(원칙 03):** 미치환 `{{…}}` 0개 (CI 게이트에서 검사).

## 7.5 resolveTone() — 톤 SSOT (단일 진실 소스)

- 톤 결정 로직은 **오직 `resolveTone()` 하나**에만 존재.
- 리포트와 프로그램이 같은 `resolveTone()`을 호출 → **톤 일치 보장(R7·P6)**.
- 톤은 5종 (예: visionary_creator, warm_connector …) × 변주 풀.

## 7.6 폴백 체계 (대원칙-B 구현)

```
응답/풀/fingerprint 있음 → 변주 합성 (고유성)
              없음       → 기존 고정 출력 (비파괴 폴백)
```
- 모든 신규 변주 함수는 **반드시 폴백 경로**를 가진다. 없으면 머지 불가.

## 8.1 [H1] 추천 확장방향 — 의미축 라벨 카드화

**문제:** "사명이 여는 길" 확장방향이 같은 구조의 여러 문장으로만 나열되어, 무엇이 어떻게 다른 확장인지 직관적으로 안 보였다.

**해법:** 각 방향 문장을 **5개 '확장 축'** 중 하나로 분류해 아이콘+축이름 카드로 표현. (문장은 그대로 — 비파괴, 라벨만 부여)

**5개 의미축 (`_directionAxis` in report.html):**
| 축 | 아이콘 | 색상 | 분류 패턴(예) |
|----|--------|------|--------------|
| 비전 정렬 | 🎯 | #7c3aed | 비전 / 한 점 / 한 방향으로 모으면 / 흩어지지 않고 / 한가운데 |
| 영역 개척 | 🧭 | #0891b2 | 아직 없는 / 카테고리 / 이름을 붙여 / 교차점 / 새 영역 / 개척 |
| 연결·전달 | 🌉 | #0d9488 | 잇는 / 이어 / 옮겨 / 전달자 / 두 세계 / 연결 / 다리 |
| 표현·확장 | 🎨 | #db2777 | 풀어 / 콘텐츠로 / 표현 / 드러내 / 보여주 |
| 구체화 | 🔨 | #ea580c | 구체화 / 형태로 / 빚어 / 만들어가 / 쌓아 / 쌓여 |

- **분류 원리:** 문자열 패턴 매칭(비파괴). 매칭 안 되면 인덱스 기반 기본 축으로 순환.
- **구현 위치:** `report.html` `_directionAxis(text, idx)`.

## 8.2 [H1-fix / [1]] 축 중복 자동 방지 (dedup) — 모든 리포트 공통 원리

**문제(사용자 지적):** 4개 방향 카드 중 '비전 정렬'이 **두 번** 나오는 경우 발생. 원인은 visionary_creator 톤의 subDirection 템플릿(index 3, "{p}의 비전을…")이 단어 "비전"을 포함 → marker 기반 필터를 우회 → 비전 정렬로 분류되어 pathLine과 충돌.

**해법(비파괴):** 엔진 데이터/문장은 그대로 두고, **report 측 라벨링에서만** 축을 유일하게 분배.

```javascript
// report.html  buildDirectionCards(dirs)
var _ALL_AXES = [비전정렬, 영역개척, 연결·전달, 표현·확장, 구체화];  // 5축
var used = {};                          // 이미 쓴 축 기록
list.forEach(function(d, i){
   var ax = _directionAxis(d, i);
   if (used[ax.name]) {                 // 같은 축 중복 → 아직 안 쓴 축으로 대체
      for (k of _ALL_AXES) if (!used[k.name]) { ax = k; break; }
   }
   used[ax.name] = true;
   // 카드 렌더
});
```

> **★ 모든 리포트 공통 원리:** "같은 구조 문장이 겹쳐도, 카드의 축 라벨만은 서로 달라야 부채꼴이 N방향으로 보인다." 검증: fp 101/103/250 모두 axis-unique=true.

## 8.5 [josa] 비전 연결어미 명사구화 (_shortVision)

**문제:** 비전이 연결어미('~며/~면서/~고/~가며/~어 가며')로 끝날 때 `'{v}'{조사}` 합성 시 `"열어 가며'가"` 같은 비문 발생.

**해법 (`report-engine-v4.js` `_shortVision(v)`):**
```javascript
// 종결어미·연결어미·끝조사를 순차 절단하여 명사구화
t = t.replace(/(으로 기억된다|로 기억된다|된다|살아간다|한다)\s*\.?$/, "");  // 종결
t = t.replace(/\s*(?:열어|만들어|되어|이루어|나아|살아)?\s*(?:가며|가면서|가고|간다|가는)\s*$/, "");
t = t.replace(/(?:며|면서|고)\s*$/, "");                                     // 일반 연결어미
t = t.replace(/(사람|이|것|길|역할|존재|주인|자리|중심|곁|편)(으로|로|을|를|...)$/, "$1");  // 끝조사
```
- 예) `"...자기 길을 열어 가며"` → `"...자기 길"` (clean).
- **과도 절단 방지:** 절단 후 남는 어절이 2글자 이상일 때만 '~으로/~로' 일반 절단.

# 제11부 · 톤 5종 상세 (Tone Variants)

> **톤(Tone) = 리포트·프로그램 전체의 인격.** 같은 응답 데이터라도 톤이 다르면 문장의 결이 달라진다.
> 톤은 **`resolveTone()` 한 곳(SSOT, 원칙 04)** 에서만 결정되며, 리포트와 프로그램이 같은 톤을 공유한다(R7·P6).
> 정의 위치: `data/report-rules.json` → `toneVariants.variants`.

## 11.1 톤 선택 규칙 (selectionRule)

> ① **Q13 핵심가치 카테고리**(관계지향/성장지향/원칙지향/자유지향)와 ② **4축 중 최고 축**을 결합해 결정.
> 동률 시 우선순위: **principled_designer > warm_connector > visionary_creator > pragmatic_achiever > reflective_explorer**

**4축 → 톤 기본 대응 (`AXIS_TO_TONE`):**
| 4축 | 대응 톤 |
|-----|---------|
| self_understanding (자기이해) | reflective_explorer |
| self_expression (자기표현) | warm_connector |
| self_design (자기설계) | principled_designer |
| self_execution (자기실행) | pragmatic_achiever |

## 11.2 정밀 가중 모델 (PR#199 — 유형화 방지)

> 가치(Q13)만으로는 흔한 '성장지향' 하나가 visionary/pragmatic을 동시에 만점으로 끌어올려 쏠림(유형화)이 생긴다. 그래서 4개 점수를 합산해 정밀 분리한다.

```
점수(tone) = vcScore + axScore + auxScore + domBonus

vcScore  = 톤의 가치카테고리가 Q13에 등장한 빈도 합 × 2.0  (일치 시 최소 2.0 보장)
axScore  = top1 축 일치 +2.0,  top2 축 일치 +1.0
auxScore = Q63 나침반(+1.0/개) · Q6 성향(+0.4/개) 보조신호 합
domBonus = 주도 카테고리가 톤 핵심가치(vc[0])와 일치 시 +2.0~5.0 (우세도 비례)
```
- **동점 해소 순서:** auxScore 큰 쪽 → vcScore 큰 쪽 → 고정 priority.
- **보조신호 출처:** `COMPASS_TO_CATEGORY`(Q63), `TRAIT_TO_CATEGORY`(Q6).

## 11.3 5종 톤 상세표

각 톤은 `label`(2단 헤드라인의 1단), `header`, `coreOneLine`, `missionTone`, `visionTone`, `executionType/Style`을 가진다. `{name}{values}{domain}{career_field}`는 응답으로 치환되는 토큰(2단 고유성).

### ① principled_designer — 원칙 중심의 분석적 설계자
| 항목 | 값 |
|------|-----|
| 트리거 | 가치=원칙지향 / 축=self_design·self_understanding |
| 핵심 한 줄 | 신념을 기반으로 구조를 설계하고 사람의 변화를 이끄는 통찰형 전략가 |
| 실행유형/스타일 | 원칙 기반 분석형 설계가 / 신념 중심형 |
| 사명 톤 | 옳다고 믿는 가치를 기준으로 사람들이 더 나은 방향으로 가도록 돕는 존재 |
| 비전 톤 | 자신만의 철학·방법론을 갖춘 설계자 → 구조적 변화를 만드는 멘토·연결자 |

### ② warm_connector — 관계 중심의 따뜻한 연결자
| 항목 | 값 |
|------|-----|
| 트리거 | 가치=관계지향 / 축=self_expression·self_understanding |
| 핵심 한 줄 | 따뜻한 공감으로 마음을 잇고, 의미 있는 관계로 변화를 만드는 연결자 |
| 실행유형/스타일 | 공감·관계형 실행가 / 감정 중심형 |
| 사명 톤 | 사람들이 안전하게 자기 자신으로 머무를 관계의 공간을 만드는 존재 |
| 비전 톤 | 사람을 깊이 이해하는 공감형 전문가 → 회복·성장 모델을 제시하는 따뜻한 리더 |

### ③ visionary_creator — 의미 중심의 비전형 창조자
| 항목 | 값 |
|------|-----|
| 트리거 | 가치=성장지향·자유지향 / 축=self_design·self_execution |
| 핵심 한 줄 | 의미와 가능성에 이끌려 새로운 방식·콘텐츠로 영향을 만드는 비전형 창조자 |
| 실행유형/스타일 | 의미·비전형 창조가 / 탐색·창조형 |
| 사명 톤 | 기존 틀을 넘어 새로운 가능성을 발견하고 사람들과 나누는 존재 |
| 비전 톤 | 자기 색깔·메시지를 가진 창조자 → 새 가능성·방향성을 제시하는 비전 리더 |

### ④ pragmatic_achiever — 결과 중심의 실행형 성취자
| 항목 | 값 |
|------|-----|
| 트리거 | 가치=성장지향·원칙지향 / 축=self_execution·self_design |
| 핵심 한 줄 | 명확한 목표·꾸준한 실행으로 결과를 만들고 성취로 신뢰를 쌓는 추진형 성취자 |
| 실행유형/스타일 | 목표·결과형 실행가 / 성과 중심형 |
| 사명 톤 | 모호한 가능성을 구체적 결과로 전환하는 존재 |
| 비전 톤 | 결과를 만들어내는 실행 전문가 → 비전을 현실로 옮기는 신뢰받는 리더·멘토 |

### ⑤ reflective_explorer — 내면 중심의 사색형 탐험가
| 항목 | 값 |
|------|-----|
| 트리거 | 가치=자유지향·관계지향 / 축=self_understanding·self_expression |
| 핵심 한 줄 | 깊은 내면 성찰과 진지한 탐구로 자신과 세상의 의미를 발견하는 사색형 탐험가 |
| 실행유형/스타일 | 성찰·탐색형 사색가 / 내적 동기형 |
| 사명 톤 | 가치를 마음의 나침반 삼아 내면의 질문에 정직하게 응답하며 길을 만드는 존재 |
| 비전 톤 | 깊은 사고·통찰을 갖춘 사색가 → 본질적 질문·새 관점을 제시하는 영감의 원천 |

## 11.4 톤별 부수 자산 (영역별 변주 풀)

각 톤은 다음 영역에서 톤 전용 변주 풀을 갖는다 (제8부·제7부 variantIdx로 fingerprint 선택):
- **교육 추천(EDU_FALLBACK):** 예) warm_connector → "코칭·퍼실리테이션 과정", visionary_creator → "스토리텔링·내러티브 훈련"
- **진로 추천(CAREER_FALLBACK):** 예) warm_connector → "커뮤니티 빌더", visionary_creator → "콘텐츠 디렉터/크리에이터"
- **고유한 결(toneTagline):** 톤×가치카테고리(관계/자유/성장/원칙) 조합별 문구 풀 — 예) `warm_connector|관계지향` → "사람의 마음을 안전한 자리에 머무르게 하는"
