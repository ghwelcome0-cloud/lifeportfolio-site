## 0.2 인바디(InBody) 비유 — 제품 구조를 이해하는 가장 쉬운 그림

```
   인바디 검사지            →   인바디 기반 맞춤 운동프로그램
   ───────────                  ──────────────────────
   인생포트폴리오 리포트     →   맞춤 실행프로그램
   (지금 나를 진단)              (그래서 무엇을 할지)
```

- **리포트 = 검사지:** 응답 데이터로 "지금 당신은 이런 사람"을 정밀하게 보여 준다.
- **프로그램 = 운동프로그램:** 그 진단을 근거로 "그래서 3주/3개월/1년 동안 이렇게 하라"를 준다.
- 둘은 **같은 톤(SSOT)** 으로 묶여야 하며, 프로그램은 반드시 리포트를 근거로 한다.

## 0.3 자산화의 큰 구조 (★ 사업의 전체 지도)

> 이것이 인생포트폴리오가 향하는 **최종 그림**이다. 리포트·프로그램은 그 첫 두 단계일 뿐이다.

```
[1] 리포트 + 맞춤 실행프로그램         ← 발견 + (살아낼) 설계
        │   "지금 나"를 진단하고, 무엇을 할지 받는다
        ▼
[2] 코칭 + 다이어리                    ← 살아냄 (실행의 지속)
        │   고객이 실제로 실행을 이어 간다. 코칭이 곁을 지키고,
        │   다이어리가 살아낸 흔적을 기록한다.
        ▼
[3] 방향별 커뮤니티                    ← 남김 + 연결성
        │   각 고객의 '방향'에 맞는 커뮤니티가 형성된다.
        │   같은 결을 가진 사람들이 서로의 자산을 연결한다.
        ▼
[4] 인생 자산화                        ← 가치 ↑ · 연결성 ↑
        한 사람의 인생이 자산이 되고,
        자산이 되면 가치가 높아지고 다른 자산과의 연결성도 높아진다.
```

**단계별 설명:**

| 단계 | 무엇을 | 성경적 대응 | 제품/기능 |
|------|--------|-------------|-----------|
| 1. 진단·설계 | 리포트 + 프로그램 | 발견 | 본 규칙서 제4·5부 |
| 2. 실행 지속 | 코칭 + 다이어리 | 살아냄 | (확장 로드맵) |
| 3. 연결 | 방향별 커뮤니티 | 남김 | (확장 로드맵) |
| 4. 자산화 | 가치·연결성 상승 | 인생 자산화 | 전체의 도달점 |

> **빌더가 기억할 것:** 리포트/프로그램을 만들 때도 항상 이 큰 구조를 의식한다. 리포트의 "다음 단계", 프로그램의 "축적 사이클(§6)"은 모두 [2]→[3]→[4]로 이어지는 **나선형 성장**의 입구다. 결과물은 닫힌 문서가 아니라 다음 사이클의 출발점이다.

# 제1부 · 미션과 철학 — 왜 만드는가

## 1.1 한 문장 미션

> **인생포트폴리오는 한 사람의 인생을 자산화한다.**
> 자산이 되면 그 사람의 가치가 높아지고, 다른 자산과의 연결성도 높아진다.

## 1.2 발견 → 살아냄 → 남김 → 인생 자산화

```
 발견          →   살아냄           →   남김            →   인생 자산화
 진단 응답         실행 프로그램         열매 / 결과물        가치 · 연결성 ↑
```

- **누구나 따라 할 수 있다. 그러나 진짜로 따라 하면 하나님 말씀대로 살고 열매를 맺는다.**
- **80억 인구 — 최소 국내 5,000만 명 — 각자의 고유성(Only One)을 보장하는 그 도달점까지 품질을 높인다.**

## 1.3 의미경제(Meaning Economy) 관점

기술(AI)이 효율을 맡는 시대에, 인간의 차별성은 **의미를 창조하는 능력**에서 나온다. 인생포트폴리오는 평범한 개인이 자기 인생의 의미를 **데이터로 발견하고, 실행으로 살아내고, 결과물로 남겨** 자산 생산자가 되도록 돕는다.

## 1.4 성경적 설계 기반

| 구절 | 의미 | 제품 반영 |
|------|------|-----------|
| 마태복음 24:45 | "때를 따라 양식을 나눠 줄 충성되고 지혜로운 종" | 각 사람에게 **맞춤(때를 따라)** 양식 — 고유성 |
| 마태복음 28:18–20 | "제자로 삼아 … 가르쳐 지키게 하라" | 발견에 그치지 않고 **살아내게(지키게)** 한다 — 프로그램·코칭 |

> **흔들릴 때 돌아올 곳:** 어떤 기능을 더할지 말지 판단이 안 설 때 → "이것이 한 사람의 고유성을 더 살리고, 자산화에 더 가까이 가게 하는가?"를 물어라. 답이 '아니오'면 하지 않는다.

## 1.5 비전 통찰 3종 (총괄, 2026-07-16 기록)

> 아래는 P24(융합 리포트) 확인 후 총괄(김영식)이 남긴 방향 통찰이다. 코드/카피가 아닌 **철학의 나침반**으로 기록한다. (코칭 가이드 등 구체 산출물은 별도 스프린트에서 착수)

**① 융합 = 방향의 확장 (닫힌 정답 → 열린 좌표)**
- 기존 직업 매칭은 "너는 A직업"이라는 **닫힌 정답**이었다. 인생포트폴리오는 무게중심 **좌표**(대원칙-C)로 표현하므로, 고객은 그 좌표를 ①지금 존재하는 직업으로 해석할 수도, ②**아직 세상에 없는, 자신만이 만들 진로**로도 해석할 수 있다.
- **방향은 주되 답은 열어둔다.** 진로·직업·교육·훈련을 융합으로 표현한 결과, 사람들의 선택 방향이 넓어졌다.

**② "절대평가" 기여 — 비교불가 기준의 제시**
- 상대평가만 있던 시절, 달란트 비유(마 25:15 "각각 그 재능대로")에서 절대평가 개념이 나왔다.
- 인생포트폴리오는 **"80억 인구 누구나 자기만의 Only One"** 이라는 **비교불가·절대적 기준**을 제시한다. 남과의 비교(상대평가)가 아니라 **자기 좌표를 발견·살아냄·남김**. → 대원칙-A(고유성)·대원칙-D(부재·비교 프레임 금지)와 한 몸.

**③ 코칭의 전환 — "직업 전문성" → "방향성·자산화"**
- 두 리포트(진단 + 실행프로그램)가 코치·고객센터에 **공통 언어**를 준다. 코치가 특정 직업 지식이 없어도 "당신의 좌표 → 이번 주 살아냄 → 누군가의 양식으로 남김"이라는 **자산화 여정**을 이끌 수 있다.
- 한국식 **"정답 강박·기존 틀 못 넘음"** 을 깨고, 진짜 고유성·맞춤화에 걸맞은 가치 창출과 자산 형성으로 **돌파하는 힘**을 주는 것이 목표.

# 제2부 · 네 대원칙 + 핵심 원칙 (헌법)

## 2.1 대원칙-A 「고유성 종합 × 직관적 단일표현 (2단 구조)」 — 최우선

> **"고유성은 종합하되, 표현은 직관적으로 딱 하나로."**
> **"종합은 디테일에서, 표현은 헤드라인에서."**

### 문제 (개선 전)
표현이 골격(고정 템플릿)에 갇혀 있었다. 같은 톤·분야 사용자끼리 **단일 고정 문구가 100% 동일** → 80억의 고유성이 표현 단계에서 사라짐. "새로고침해도 같은 결과"로 체감.

### 해법 (2단/3단 구조)
| 단 | 역할 | 성격 |
|----|------|------|
| **1단 헤드라인** | 직관 압축 (브랜드 슬로건 수준) | 누가 봐도 즉시 이해 |
| **2단 디테일/서브라인** | 응답 종합 고유성 | 같은 톤이라도 응답마다 다르게 |
| **3단 푸터** | 근거 Q번호 (선택) | 신뢰의 근거 |

### 2단 구조 적용 영역 (현황)
| 영역 | 1단 헤드라인(직관) | 2단 디테일(응답 종합 고유성) |
|------|-------------------|------------------------------|
| 사명/비전 | missionHeadline · visionHeadline | 응답 직합성 (Q13·Q63·Q75·Q73) |
| 고유한 결 | toneLabel (약 2,300종) | toneTagline 변주 8종 (variantIdx 0x7A6C) |
| 사명이 여는 길 | CareerEngine newPaths | 응답 기반 직업 매칭 |
| 분기 테마 | Quarter Heading (톤×Compass) | subline 8종 (variantIdx 0x5131) |
| 3주 루틴 | 주차 제목 | 주차 의미별 subline 6종×3 (0x6201+i) |
| 다음 단계 | "이 사이클을 마치면" | 변주 6종+도입 4종 (0x7F11/0x7F23) |

> **모든 2단은 fingerprint(=hash(응답)) 기반 결정 선택** → 같은 응답=같은 결과(재현성), 응답 부재 시 기존 출력 폴백(대원칙-B).

## 2.2 대원칙-B 「축적 원칙」 — Trade-off 금지

> **"개선은 누적한다. 어제의 사용자가 본 결과는 오늘도 유효해야 하고, 오늘의 개선은 내일 더 정교해진다."**

| 항목 | 내용 |
|------|------|
| **B-1 폴백 보존** | 풀 부재 / fingerprint 없음 → 기존 출력 그대로. 새 기능이 옛 기능을 깨지 않는다. |
| **B-2 재현성** | 같은 응답 = 같은 결과. fingerprint 결정 선택. 같은 fingerprint 2회 → 동일 산출(회귀 테스트 의무). |
| **B-3 버전 누적** | 옛 규칙 삭제 금지. v1.0→v1.2→v1.4→v2.0. 구식 로직은 dead code로 보존(롤백 대비). |
| **검증 게이트(4중)** | ① 폴백 ② 재현성 ③ 회귀 테스트 ④ 버전 누적 — 모두 통과해야 머지. |

## 2.2.1 대원칙-C 「융합 표현 원칙」 — 나열 금지, 무게중심 복원 (2026-07-15 신설)

> **"여러 응답 요소는 나열하지 않고 융합한다. 융합 = 속성 벡터들의 무게중심(centroid) + 그 좌표를 사람 말로 복원(decode)."**

고객이 여러 요소(관심 분야 Q75, 강점, 판단기준 등 복수 응답)를 선택했을 때, 그것을
`A·B·C`처럼 **나열(enumeration)** 하거나 `A와 B, C를 잇는`처럼 **연결어로 이어 붙이면 안 된다.**
원소가 그대로 보존되므로 "융합"이 아니라 "목록"이며, 고객이 자신만의 고유한 정체성으로
직관 이해할 수 없다.

| 항목 | 내용 |
|------|------|
| **C-1 나열 금지** | `label(A)+"·"+label(B)` 형태 금지. 원분야/원요소 단어를 그대로 이어 붙이지 않는다(§7과 연동). |
| **C-2 벡터 분해** | 각 요소를 의미 **속성 벡터**로 사전 정의한다. (예: 분야 3축 — CORE 무엇을 / ACT 어떻게 / FRUIT 무엇으로) |
| **C-3 무게중심 합성** | 선택된 N개 요소의 벡터를 **가중 합**(선택 순서 = 가중치, 첫 요소 최대)하여 **무게중심 좌표**를 구한다. word2vec식 additive compositionality("king−man+woman≈queen") 원리. |
| **C-4 복원(decode)** | 무게중심 좌표에서 각 축의 **argmax 어휘**를 골라 **하나의 새 정체성 문장**으로 복원한다. 원단어는 좌표 연산에 흡수되어 **사라진다**(§7 자동 준수). |
| **C-5 결정론 tie-break** | 동점(tie)은 **fingerprint 해시**로 가른다 → 재현성(대원칙-B) + 고유성(대원칙-A) 동시 확보. `Math.random` 금지. |
| **C-6 개수 적응** | 1개/2개/3+개 선택에 따라 문장 골격을 달리해 **모든 조합**을 자연스럽게 표현한다. |
| **C-7 원분야 무손상 (raw domain) ★2026-07-16 신설** | 융합 좌표 연산의 **입력은 정규화(alias)하지 않은 원분야 그대로** 사용한다. `domainAliases`(경영→경제, 건강→의료 등)는 **오직 진로 풀(domainPools) 사전 조회 경로에서만** 쓰고, 벡터 분해(C-2)에는 **`DOMAIN_ATTR_KO`(21개 원분야 속성 사전)를 직접** 참조한다. alias를 거치면 서로 다른 원분야가 같은 좌표로 뭉개져(예: 경영→경제로 core가 `가치`로 바뀜) **고객별 순서 민감 고유성이 손실**된다. 융합 입력 배열은 `trim → DOMAIN_ATTR_KO 존재 필터`만 적용하고 alias는 우회한다. |
| **C-8 짧은 명사구 복원 ★2026-07-16 신설** | 복원(C-4) 결과가 이중 관형절(예: `~키워 내는 ~기르는 눈`)이나 동일 연결어 반복(`잇는 … 잇는`)이 되면 **비문**이다. 장기·교육 문형은 **짧은 명사 풀(예: 멀리 보는 안목 / 깊이 있는 통찰 / 크게 보는 눈)** 로 마감하고, 하나의 문장에 같은 연결어(`잇는` 등)를 2회 이상 쓰지 않는다. |
| **적용 범위** | 리포트 엔진(report-engine-v4) · 맞춤 실행프로그램(program-engine) · 진로/교육 엔진(career-engine) **모두**. 각 고객 응답이 종합·융합되어 반영. |

> **예시:** 종교(신념/지킴) + 교육(가르침/성장) + 경영(질서/성과)
> → centroid ≈ {가치↑, 전달↑, 구조화↑}
> → decode: **"신념을 가르쳐 조직으로 키우는 일"** (원단어 소멸, 제4의 정체성)
>
> ❌ `"종교·교육·경영이 만나는 자리"`(나열) · `"종교와 교육, 경영을 잇는"`(연결어 나열) — 모두 금지.
>
> 상세 설계: `docs/청사진_P20_분야융합엔진.md`

## 2.2.2 대원칙-D 「정체성 뼈대 카피 원칙」 — 전 페이지 공통 (2026-07-16 신설)

> **"모든 페이지의 카피는 핵심 정체성 `발견 → 살아냄 → 양식(남김) → 자산화`를 뼈대로 삼고, 각 부위·기능에 따라 살을 붙인다. 뼈대는 불변, 살은 맥락에 맞게 변주."**

랜딩·로그인·검사·리포트·프로그램 등 **어느 화면의 어떤 문구를 쓰거나 고칠 때에도** 아래 5원칙을 따른다.
서비스가 일관된 가치를 전달하려면, 문구가 제각각이 아니라 **하나의 정체성에서 파생**되어야 한다.

| 항목 | 규칙 | 근거 |
|------|------|------|
| **D-1 긍정형 유일함** | 유일함은 **"당신에게 있다"(존재)** 로 표현한다. **"아무에게도 없다"(부재)** 로 쓰지 않는다. | CliftonStrengths 공식 카피 *"talents that make you different from anyone else"* (부재가 아닌 태생적 고유함) |
| **D-2 경쟁자·부재 프레임 금지** | 타 서비스/유형론을 **부정·저격하지 않는다.** ❌`유형으로 분류된 결과가 아니라` ❌`MBTI보다 정확` ❌`그 누구에게도 존재하지 않습니다`. 자기 가치만 말한다. | 윤리 가드레일(두려움/저격 지양) · 벤치마킹(16P·Gallup 모두 경쟁자 언급 0) |
| **D-3 과잉 설명 금지** | 우리가 "왜 특별한지" 논증하지 않는다. 고객이 받는 **고유 리포트 자체**가 증거다. 필요한 만큼만 전한다("Less is more"). | 모바일 첫 화면 · "말이 적을수록 고객이 스스로 판단" |
| **D-4 정체성 3단 필수 연결** | 핵심 카피는 **발견 → 살아냄 → 양식(남김)** 흐름과, 북극성 **"삶이 자산이 된다"** 에 반드시 연결한다. 서비스 공용 표기 `발견 · 살아냄 · 양식` 어휘를 사용해 전 페이지 용어를 통일한다. | 북극성 D2 카피 · card3/eyebrow 등 기존 전 페이지 표기 |
| **D-5 분량·행동 유도** | 핵심 배너/헤드라인은 **1~2문장**. 설명 대신 **행동·여정 동사**(발견하다·살아내다·남다)로 연다. | 벤치마킹(Gallup *"Discover…"*) |

> **금지 표현 예 (전 페이지):** ❌"인생이 바뀐다" ❌"MBTI보다 정확하다" ❌"전문 임상 진단" ❌"그 누구에게도 존재하지 않습니다" ❌"유형으로 분류된 결과가 아니라"
>
> **적용 실사례 (2026-07-16, report Only One 배너):**
> - Before: "…같은 리포트는 **그 누구에게도 존재하지 않습니다. 유형으로 분류된 결과가 아니라**, …첫 행동이 담긴 살아있는 인생 포트폴리오입니다." (부재+저격+과잉설명, 3문장)
> - After: "76문항에 담긴 **당신의 이야기**에서 사명과 강점을 **발견**하고, 그 발견을 **살아내며**, 살아낸 하루가 **누군가의 양식으로 남습니다.** 그렇게 당신의 삶이 자산이 됩니다." (긍정+3단+자산 연결, 2문장)
> - chips: `76문항으로 완성` · `발견 → 살아냄 → 양식` · `당신만을 위한 한 권`

# 제7부 · 합성 엔진 구현 디테일

> **엔진은 규칙 기반 합성(NOT LLM).** 같은 응답엔 항상 같은 답 — 신뢰 가능한 진단. 이것이 9,900원 진단의 품질 보증 근거다.

## 7.1 fingerprint — 고유성의 씨앗

```
fingerprint = hash(응답 전체)
```
- 같은 응답 → 같은 fingerprint → 같은 결과(재현성, B-2).
- 응답이 없으면 fingerprint 없음 → 변주 idx 0 반환(회귀 안전, B-1).

## 7.2 variantIdx(salt) — 결정적 변주 선택

> 같은 톤·분야라도 응답마다 다른 문장을 고르되, 같은 응답엔 항상 같은 문장.

```
variantIdx(salt):
   x = fingerprint XOR (salt × 0x9E3779B1)   // 황금비 상수로 bit-mixing
   x = xorshift(x)                            // 비트 섞기
   return x % pool.length                     // 풀에서 결정적 선택
   (fingerprint 없음 → 0 반환)
```

**salt 배정표 (영역별 고유 salt — 충돌 방지):**
| salt | 영역 |
|------|------|
| 0x7A6C | 고유한 결 toneTagline |
| 0x5131 | 분기 테마 subline |
| 0x6201+i | 3주 루틴 주차별 subline |
| 0x7F11 | 다음 단계 첫 항목 |
| 0x7F23 | 다음 단계 둘째 항목 도입 |

## 7.3 _fixJosaPairs — 조사 자동 보정

> 한국어 자연성(원칙 05). 받침 유무에 따라 조사를 자동 선택.

대상 조사쌍: **을/를 · 이/가 · 와/과 · 은/는 · 으로/로**

```
_jongBefore(text, pos):
   따옴표·괄호를 건너뛰고 직전 '한글 음절'을 찾아 받침 유무 판정
```
- 예) `'통찰력'으로` ✓ / `'아이디어'로` ✓
- 자모 가드(R10): `더하ㄴ다 → 더한다`, `창조와 자유이 → 자유가`.

## 7.4 tpl() — 토큰 치환

- compassKw·primaryDomain·userTopStrength 등은 **토큰**으로 보관 → 출력 직전 `tpl()`로 일괄 치환.
- **완결성(원칙 03):** 미치환 `{{…}}` 0개 (CI 게이트에서 검사).

## 7.5 resolveTone() — 톤 SSOT (단일 진실 소스)

- 톤 결정 로직은 **오직 `resolveTone()` 하나**에만 존재.
- 리포트와 프로그램이 같은 `resolveTone()`을 호출 → **톤 일치 보장(R7·P6)**.
- 톤은 5종 (예: visionary_creator, warm_connector …) × 변주 풀.

## 7.6 폴백 체계 (대원칙-B 구현)

```
응답/풀/fingerprint 있음 → 변주 합성 (고유성)
              없음       → 기존 고정 출력 (비파괴 폴백)
```
- 모든 신규 변주 함수는 **반드시 폴백 경로**를 가진다. 없으면 머지 불가.

## 8.5 [josa] 비전 연결어미 명사구화 (_shortVision)

**문제:** 비전이 연결어미('~며/~면서/~고/~가며/~어 가며')로 끝날 때 `'{v}'{조사}` 합성 시 `"열어 가며'가"` 같은 비문 발생.

```javascript
// 종결어미·연결어미·끝조사를 순차 절단하여 명사구화
t = t.replace(/(으로 기억된다|로 기억된다|된다|살아간다|한다)\s*\.?$/, "");  // 종결
t = t.replace(/\s*(?:열어|만들어|되어|이루어|나아|살아)?\s*(?:가며|가면서|가고|간다|가는)\s*$/, "");
t = t.replace(/(?:며|면서|고)\s*$/, "");                                     // 일반 연결어미
t = t.replace(/(사람|이|것|길|역할|존재|주인|자리|중심|곁|편)(으로|로|을|를|...)$/, "$1");  // 끝조사
```
- 예) `"...자기 길을 열어 가며"` → `"...자기 길"` (clean).
- **과도 절단 방지:** 절단 후 남는 어절이 2글자 이상일 때만 '~으로/~로' 일반 절단.

# 제11부 · 톤 5종 상세 (Tone Variants)

> **톤(Tone) = 리포트·프로그램 전체의 인격.** 같은 응답 데이터라도 톤이 다르면 문장의 결이 달라진다.
> 톤은 **`resolveTone()` 한 곳(SSOT, 원칙 04)** 에서만 결정되며, 리포트와 프로그램이 같은 톤을 공유한다(R7·P6).
> 정의 위치: `data/report-rules.json` → `toneVariants.variants`.

## 11.1 톤 선택 규칙 (selectionRule)

> ① **Q13 핵심가치 카테고리**(관계지향/성장지향/원칙지향/자유지향)와 ② **4축 중 최고 축**을 결합해 결정.
> 동률 시 우선순위: **principled_designer > warm_connector > visionary_creator > pragmatic_achiever > reflective_explorer**

**4축 → 톤 기본 대응 (`AXIS_TO_TONE`):**
| 4축 | 대응 톤 |
|-----|---------|
| self_understanding (자기이해) | reflective_explorer |
| self_expression (자기표현) | warm_connector |
| self_design (자기설계) | principled_designer |
| self_execution (자기실행) | pragmatic_achiever |

## 11.2 정밀 가중 모델 (PR#199 — 유형화 방지)

> 가치(Q13)만으로는 흔한 '성장지향' 하나가 visionary/pragmatic을 동시에 만점으로 끌어올려 쏠림(유형화)이 생긴다. 그래서 4개 점수를 합산해 정밀 분리한다.

```
점수(tone) = vcScore + axScore + auxScore + domBonus

vcScore  = 톤의 가치카테고리가 Q13에 등장한 빈도 합 × 2.0  (일치 시 최소 2.0 보장)
axScore  = top1 축 일치 +2.0,  top2 축 일치 +1.0
auxScore = Q63 나침반(+1.0/개) · Q6 성향(+0.4/개) 보조신호 합
domBonus = 주도 카테고리가 톤 핵심가치(vc[0])와 일치 시 +2.0~5.0 (우세도 비례)
```
- **동점 해소 순서:** auxScore 큰 쪽 → vcScore 큰 쪽 → 고정 priority.
- **보조신호 출처:** `COMPASS_TO_CATEGORY`(Q63), `TRAIT_TO_CATEGORY`(Q6).

## 11.3 5종 톤 상세표

각 톤은 `label`(2단 헤드라인의 1단), `header`, `coreOneLine`, `missionTone`, `visionTone`, `executionType/Style`을 가진다. `{name}{values}{domain}{career_field}`는 응답으로 치환되는 토큰(2단 고유성).

### ① principled_designer — 원칙 중심의 분석적 설계자
| 항목 | 값 |
|------|-----|
| 트리거 | 가치=원칙지향 / 축=self_design·self_understanding |
| 핵심 한 줄 | 신념을 기반으로 구조를 설계하고 사람의 변화를 이끄는 통찰형 전략가 |
| 실행유형/스타일 | 원칙 기반 분석형 설계가 / 신념 중심형 |
| 사명 톤 | 옳다고 믿는 가치를 기준으로 사람들이 더 나은 방향으로 가도록 돕는 존재 |
| 비전 톤 | 자신만의 철학·방법론을 갖춘 설계자 → 구조적 변화를 만드는 멘토·연결자 |

### ② warm_connector — 관계 중심의 따뜻한 연결자
| 항목 | 값 |
|------|-----|
| 트리거 | 가치=관계지향 / 축=self_expression·self_understanding |
| 핵심 한 줄 | 따뜻한 공감으로 마음을 잇고, 의미 있는 관계로 변화를 만드는 연결자 |
| 실행유형/스타일 | 공감·관계형 실행가 / 감정 중심형 |
| 사명 톤 | 사람들이 안전하게 자기 자신으로 머무를 관계의 공간을 만드는 존재 |
| 비전 톤 | 사람을 깊이 이해하는 공감형 전문가 → 회복·성장 모델을 제시하는 따뜻한 리더 |

### ③ visionary_creator — 의미 중심의 비전형 창조자
| 항목 | 값 |
|------|-----|
| 트리거 | 가치=성장지향·자유지향 / 축=self_design·self_execution |
| 핵심 한 줄 | 의미와 가능성에 이끌려 새로운 방식·콘텐츠로 영향을 만드는 비전형 창조자 |
| 실행유형/스타일 | 의미·비전형 창조가 / 탐색·창조형 |
| 사명 톤 | 기존 틀을 넘어 새로운 가능성을 발견하고 사람들과 나누는 존재 |
| 비전 톤 | 자기 색깔·메시지를 가진 창조자 → 새 가능성·방향성을 제시하는 비전 리더 |

### ④ pragmatic_achiever — 결과 중심의 실행형 성취자
| 항목 | 값 |
|------|-----|
| 트리거 | 가치=성장지향·원칙지향 / 축=self_execution·self_design |
| 핵심 한 줄 | 명확한 목표·꾸준한 실행으로 결과를 만들고 성취로 신뢰를 쌓는 추진형 성취자 |
| 실행유형/스타일 | 목표·결과형 실행가 / 성과 중심형 |
| 사명 톤 | 모호한 가능성을 구체적 결과로 전환하는 존재 |
| 비전 톤 | 결과를 만들어내는 실행 전문가 → 비전을 현실로 옮기는 신뢰받는 리더·멘토 |

### ⑤ reflective_explorer — 내면 중심의 사색형 탐험가
| 항목 | 값 |
|------|-----|
| 트리거 | 가치=자유지향·관계지향 / 축=self_understanding·self_expression |
| 핵심 한 줄 | 깊은 내면 성찰과 진지한 탐구로 자신과 세상의 의미를 발견하는 사색형 탐험가 |
| 실행유형/스타일 | 성찰·탐색형 사색가 / 내적 동기형 |
| 사명 톤 | 가치를 마음의 나침반 삼아 내면의 질문에 정직하게 응답하며 길을 만드는 존재 |
| 비전 톤 | 깊은 사고·통찰을 갖춘 사색가 → 본질적 질문·새 관점을 제시하는 영감의 원천 |

## 11.4 톤별 부수 자산 (영역별 변주 풀)

각 톤은 다음 영역에서 톤 전용 변주 풀을 갖는다 (제8부·제7부 variantIdx로 fingerprint 선택):
- **교육 추천(EDU_FALLBACK):** 예) warm_connector → "코칭·퍼실리테이션 과정", visionary_creator → "스토리텔링·내러티브 훈련"
- **진로 추천(CAREER_FALLBACK):** 예) warm_connector → "커뮤니티 빌더", visionary_creator → "콘텐츠 디렉터/크리에이터"
- **고유한 결(toneTagline):** 톤×가치카테고리(관계/자유/성장/원칙) 조합별 문구 풀 — 예) `warm_connector|관계지향` → "사람의 마음을 안전한 자리에 머무르게 하는"

> **R5 잔존 어휘 0:** visionary_creator 등에서 '마음' 같은 톤 누수 리터럴은 변수화로 0건 처리(PR#63).

```
   ├─ Q75(분야) + Q77(활동) + Q1(직무) → CareerEngine → 진로 3종
   ├─ Q13·Q63·Q75·Q73 종합 → 사명/비전 직합성(R8, 7-slot 골격 폐기)
   └─ 전체 응답 hash → fingerprint → variantIdx(salt) → 2단 변주(고유성)
                                              ↓
                        리포트(10단) + 프로그램(7섹션)  ← 같은 톤(R7)
```

## E. 수동 재현 부록 (엔진 없이 사람이 손으로 제작)

> **이 부록의 목적.** 본문 제0~12부는 "왜·무엇을·어떻게"의 **스펙**이다. 그러나 실제 엔진은 약 9,000줄(report 6,559 + program 2,449)의 코드와, 응답을 해시(fingerprint)해서 변주 풀에서 문장을 **결정론적으로** 고르는 로직을 갖는다. 따라서 **"그 사람이 받은 리포트의 토씨까지 동일하게"** 는 엔진을 돌려야만 재현된다.
>
> 이 부록 E는 그 간극을 메운다. **엔진이 없어도 사람이(빌더·제자가) 손으로 같은 품질·같은 방향·같은 톤의 리포트를 작성**할 수 있도록, ① 따라 쓰는 워크시트(E-1)와 ② 엔진 속 실제 문장 후보 풀(E-2)을 수록한다.
>
> **재현 범위 명시:**
> - ✅ **방향·구조·톤·품질 판정** — 수동으로 100% 재현 가능
> - ✅ **문장 후보 선택** — E-2 풀에서 사람이 직접 고르면 동등 품질
> - ⚠️ **글자 단위 동일 결과** — 해시 연산이라 엔진 전용(수동 불가). 단, 품질은 동등.

### E-1. 수동 작성 워크시트 (Manual Build Worksheet)

> **사용법.** 한 사람의 56문항 응답을 받아 아래 5단계를 순서대로 채운다. 각 단계는 본문의 어느 부를 근거로 하는지 표기했다.

#### STEP 0 — 응답 수집 (근거: 제3부·제12부)

| 입력 | Q번호 | 받은 값 적기 |
|------|-------|--------------|
| 가치 (최대 3) | **Q13** | ____________ |
| 성향 (3개) | **Q6** | ____________ |
| 열정 (최대 2) | **Q41** | ____________ |
| 선택 기준 (최대 2) | **Q63** | ____________ |
| 성취 순간 | **Q73** | ____________ |
| 관심 분야 (최대 3) | **Q75** | ____________ |
| 활동 유형 (최대 2) | **Q77** | ____________ |
| 4축 점수 (Likert 합) | self_understanding / self_expression / self_design / self_execution | __ / __ / __ / __ |

#### STEP 1 — 톤 결정 (근거: 제11부)

1. **Q13 가치**의 최빈 카테고리를 4분류로 환산한다 → 원칙지향 / 관계지향 / 성장지향 / 자유지향
   - 환산표: 원칙지향→`principled_designer`, 관계지향→`warm_connector`, 성장지향→`visionary_creator`, 자유지향→`reflective_explorer`
2. **4축 최고점**을 보조로 본다(`AXIS_TO_TONE`): self_understanding→reflective_explorer, self_expression→warm_connector, self_design→principled_designer, self_execution→pragmatic_achiever
3. 가치 카테고리(주) × 4축(보조)이 **충돌하면 가치(Q13) 우선**. **동률이면 우선순위**로 결정:
   `principled_designer > warm_connector > visionary_creator > pragmatic_achiever > reflective_explorer`

→ **결정된 톤: ________________** (5종 중 1)

#### STEP 2 — 주 카테고리 결정 (근거: 제11부 변주 키)

- Q13의 주 카테고리(원칙/관계/성장/자유지향) 1개를 확정 → **주 카테고리: __________**
- 이 「톤 × 주 카테고리」 조합 키(예: `warm_connector|성장지향`)가 **E-2 변주 풀의 검색 키**가 된다.

#### STEP 3 — 리포트 10단 채우기 (근거: 제4부)

각 단을 본문 제4부 규칙대로 작성. **1단 헤드라인(직관) + 2단 디테일(응답 종합) + 3단 푸터(근거 Q번호)** 의 2단 구조를 모든 단에 적용.

| 단 | 내용 | 작성 시 근거 응답 |
|----|------|-------------------|
| 1 | 핵심 정체성 한 줄 | E-2의 톤×카테고리 변주구 + Q13/Q6 |
| 2 | 강점 요약 | Q6 성향 + Q41 열정 |
| 3 | 가치·신념 | Q13 |
| 4 | 동기·열정 | Q41 |
| 5 | 의사결정 방식 | Q63 |
| 6 | 성취 패턴 | Q73 |
| 7 | 관심 영역 | Q75(13영역) |
| 8 | 활동 성향 | Q77 |
| 9 | 방향 카드(여러 갈래) | 제6부 진로엔진 + DIRECTIONS 풀 |
| 10 | 다음 단계(브리지) | 프로그램으로 연결(제5부) |

> **2단 구조 자가검사:** 모든 단에 ① 직관적 한 줄, ② 응답 근거 디테일, ③ Q번호 푸터가 있는가?

- 작성 후 **4중 검증 게이트**(부록 C 체크리스트)로 합/불 판정.

### E-3. 결정성(determinism)의 원리 — 왜 글자 단위는 엔진 전용인가

> **수동 작성자가 알아야 할 단 하나의 사실.**

엔진은 56문항 응답 전체를 **djb2 계열 해시**로 32bit 정수 지문(fingerprint)을 만든다:

```
h = 5381
for each (정렬된 Q, 응답 s):
    for each char:  h = ((h << 5) + h + charCode + (idx+1)*17) | 0
fingerprint = abs(h)
```

그리고 변주 풀에서 `arr[ abs(hash) % arr.length ]`로 후보를 고른다(`pickByHash`). salt(0x7A6C·0x5131·0x6201+i·0x7F11·0x7F23)는 같은 응답이라도 단별로 다른 인덱스를 뽑게 하는 **자리 섞기 상수**다.

- **그래서:** 같은 응답 → 항상 같은 fingerprint → 항상 같은 문장 (재현성, 대원칙-B-2)
- **그래서:** 사람이 손으로는 이 32bit 연산을 똑같이 못 한다 → **글자 단위 동일은 엔진 전용**
- **그러나:** E-2 풀에서 빌더가 응답의 결에 맞는 후보를 고르면 **품질·방향·톤은 동등**하다. 엔진은 "공정한 자동 선택자"일 뿐, 후보의 품질은 모두 동일하게 검수돼 있다.

> **제자에게:** 엔진은 너의 손을 빠르고 공정하게 대신할 뿐이다. **무엇을·왜·어떤 톤으로** 쓰는지는 이 규칙서가 전부 담고 있다. E-1로 쓰고, E-2에서 고르고, 부록 C로 검증하라. 그러면 엔진 없이도 같은 인생포트폴리오를 만들 수 있다.
